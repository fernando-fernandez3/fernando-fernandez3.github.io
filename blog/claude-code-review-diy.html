Anthropic Launched a $25 PR Review Agent. I've Been Running the Same Thing for $0.02.

Fernando Fernandez 6 min read 1,150 words

Anthropic announced Code Review for Claude Code today. It's a multi-agent system that reviews your pull requests for bugs before a human reviewer sees them. Their internal tests showed that for large PRs, 84% of reviews caught something real. Human developers rejected less than 1% of Claude's findings.

That's a good product. The results are genuinely impressive.

The price is $15–$25 per review. It requires a Teams or Enterprise plan. If you're running Claude Code under the individual subscription, you can't access it.

I've been running a version of this for months. Not exactly the same thing: Anthropic's product is specialized for code diffs and GitHub integration. But the underlying pattern is identical: one agent produces output, a second agent reviews it, the first agent revises if needed. Same logic, different wrapper.

My cost per review: about $0.02.

What Anthropic's Code Review Actually Is

The launch covers a specific use case: you open a PR, Claude Code spawns a review agent, that agent reads the diff and flags bugs, unclear logic, and potential issues. According to Anthropic's internal data, for PRs with more than 1,000 changed lines, the agent finds an average of 7.5 issues. For small PRs under 50 lines, 31% still get at least one comment.

The "multi-agent" part is that Code Review isn't a single prompt. It's multiple agents working on different parts of the review in parallel, then consolidating findings. That's why the quality is higher than a simple "review this code" prompt. The parallelization lets agents focus on specific concerns.

That architecture works. I know it works because I built a simpler version of it and watched my content quality improve.

The Pattern Is Simple

You don't need the full Anthropic Code Review implementation to get value from multi-agent review. The core pattern takes about an hour to set up:

  1. Agent A produces output and writes it to a file
  2. Agent B reads that file and answers: does this actually do what was asked? What's weak or missing? Pass or revise?
  3. If revise: Agent B writes specific notes to a feedback file
  4. Agent A reads the feedback and produces a revised version
  5. Cap revisions at 2 to avoid infinite loops

That's it. I have this running for my content pipeline. My builder agent drafts, my QA agent reviews, and the builder revises if needed. The QA agent runs at :15 past the hour so the builder always finishes first.

What changed after I added it: the "yes-man problem" went away. Before, my agents produced technically-correct output that missed the point. Adequate every time, never excellent. The QA agent fixes that because it has no investment in the output. It's not trying to complete the task. Its only job is to find problems.

What I Actually Run

My setup isn't for code review. I write content, run research agents, and build guides. But the review architecture is the same:

Builder agent prompt (simplified):

Read the task file at tasks/current.md.
Produce the output. Write to output/draft.md.
If feedback exists at output/feedback.md from a previous run, incorporate it before writing.

QA agent prompt (simplified):

Read output/draft.md and tasks/current.md.
Answer:
1. Does the output address the actual task? (Yes / Partially / No)
2. What's weak, incomplete, or off?
3. Decision: PASS or REVISE

If REVISE, write specific notes to output/feedback.md.
If PASS, write APPROVED to output/feedback.md and stop.

The builder runs at :00. The QA agent runs at :15. The builder checks for feedback at :30: if feedback says REVISE, it runs again. Max two revision cycles.

That's the whole thing. No framework. No orchestration library. Two cron jobs and two markdown files.

The Part Anthropic's Product Does Better

I want to be clear about where the $25 version wins.

GitHub integration is the big one. Anthropic's Code Review attaches directly to pull requests, posts review comments inline, and integrates with existing code review workflows. My DIY setup would require writing a GitHub webhook handler, parsing diffs, formatting output as PR comments, and handling the auth flow. That's a few hundred lines of plumbing I haven't built.

Parallel sub-agents are the other thing. Their product spawns multiple specialized agents simultaneously: one looking at security, one at logic, one at edge cases. My sequential setup can do this too but requires more prompt engineering and more tokens per run.

For a dev team that already pays for Claude Teams, Code Review probably makes sense. The workflow integration alone is worth something.

For a solo builder who wants the review concept without the enterprise price tag, the pattern is what matters, not the product.

Why This Launch Matters Beyond the Price

The bigger story isn't the cost. It's that Anthropic just validated the multi-agent review pattern publicly with production data.

When I built my QA agent setup, I wasn't sure it would work. The reasoning was sound: a second independent perspective should catch things the first agent misses: but I didn't have numbers. Now Anthropic is showing 84% catch rates on large reviews and a sub-1% false positive rate from human reviewers. That's the same mechanism I'm using for content.

Multi-agent review isn't a novelty. It's how quality control should work in any agent pipeline. The only question is whether you pay $25/review for a managed version or spend an afternoon wiring up the pattern yourself.

If you're running any kind of agent pipeline that produces output someone depends on, adding a QA agent is worth the time. The "adequate every time" problem is real, and the fix isn't complicated.
