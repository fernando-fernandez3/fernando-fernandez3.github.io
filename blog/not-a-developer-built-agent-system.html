I Was an Auto Mechanic. Now I Run 10 Background AI Agents. Here's What It Actually Took.

Fernando Fernandez 8 min read 1,459 words
By Fernando Fernandez

Eight years ago I was diagnosing engine misfires under a lift in San Antonio. Now I have 10 AI agents running cron jobs at 6am every morning while I'm still asleep. One of them monitors my pool chemistry. One writes me a daily briefing. One checks that the others are still working.

I did not go to school for this.

I'm a Staff Support Engineer at Tesla. Self-taught Python. ASE Master Technician first, code second. I bring this up not because it's some inspirational story but because most of the content about Claude Code and AI agents assumes you already know what a daemon is. I didn't. I had to figure out a lot of things the slow way, and I think that's actually useful context if you're in a similar spot.

The Pragmatic Engineer just published their 2026 AI tooling survey. 95% of engineers use AI weekly. 55% now run AI agents regularly. Claude Code is #1. And agent users are "twice as excited about AI as non-users."

That gap, the 45% who haven't made the jump, is where I was 18 months ago. This is what crossing it actually looked like.

What I was trying to solve

My job generates a lot of context I need to stay on top of. Tool releases, diagnostic patterns, failure reports, support ticket trends. I was spending 30 minutes every morning just getting up to speed. Reading emails, scanning Slack, checking dashboards. It's not hard work, but it's the kind of work that drains you before the real work starts.

I also had side projects that were always "almost running." Pool automation. A call screener. A morning briefing. Each one needed attention I kept not giving it because the setup was annoying and fragile.

What I wanted was something that ran before I woke up, gave me what I needed, and didn't ask me to babysit it.

That's not a chatbot. A chatbot needs you there. What I wanted was an agent.

The thing nobody told me about agents

When I first heard "AI agent," I thought it meant some complicated autonomous system with memory, planning, tool calls, a vector database, and a dozen moving parts.

Sometimes it does mean that. Most of the time it just means: a script that runs Claude with instructions, does something with the output, and exits cleanly.

My morning briefing agent is 47 lines of Python. It reads three files, calls Claude Haiku with a prompt, and sends me a Telegram message. That's it. That's an agent.

The "architecture" that makes it actually useful is in the supporting files, not the code:

  • MEMORY.md: what the agent knows about me and my context
  • USER.md: preferences, priorities, current projects
  • AGENTS.md: what this specific agent is supposed to do

Claude reads those files at the start of every run. That's how it knows my name, my job, what I care about, and what "good output" looks like. It's not magic. It's just giving the model context it would otherwise have to ask for.

I didn't realize how much of "building an AI system" was just writing good files until I did it.

What I actually built

Ten cron jobs. Here's the breakdown:

6:00am: Morning briefing (Claude Sonnet)
Reads my calendar, recent notes, and memory files. Sends a Telegram message with what's on today and anything that needs attention. This one replaced my 30-minute morning scroll.

6:15am: Memory maintenance (Claude Haiku)
Reads my daily notes from the past week, looks for things worth keeping long-term, and updates MEMORY.md. The goal is that the other agents get smarter about me over time without me doing anything.

9:00am: Health check (Claude Haiku)
Checks that all the other agents ran, their output files are fresh, and nothing is stuck or silently broken. Alerts me if something's wrong. This is the one that saved me three times in the first month.

Noon: Pool chemistry check (Claude Haiku)
Reads the sensor data from my ESPHome setup, compares it against target ranges, and sends me an alert if pH or ORP is out of range. Before this I was doing it manually and missing it for days at a time.

6:00pm: End of day summary (Claude Haiku)
Looks at what I actually did vs. what was planned. Short. Just a few lines. Useful mostly for my own accountability.

The other five are more specific to my work and side projects. The point is that none of them are complicated. The most complex one is probably 120 lines.

Total daily cost: about $3. Most of that is the morning briefing because Sonnet is more expensive. The rest run on Haiku and cost almost nothing.

What actually took time to figure out

Memory. The first version of my system had no memory at all. Every agent woke up knowing nothing about me, my projects, or what had happened yesterday. The output was fine in isolation but useless over time. Adding the three-file memory system (MEMORY.md, USER.md, daily notes) made everything significantly more useful, and it took me an embarrassingly long time to figure out the right write discipline to make it work.

Model routing. I started by using Claude Sonnet for everything because it seemed like the "good" model. My bill was $40/day. Routing quick checks to Haiku and only using Sonnet for the complex stuff brought it down to $3. Same output quality for 95% of tasks.

Silent failures. An agent that crashes doesn't announce itself. It just doesn't run. I had a memory maintenance job silently failing for two weeks before I noticed because the morning briefing still looked normal. The health check agent fixed this. Every production system needs something watching the watchers.

Cron timing. I ran all ten jobs at 6am for the first week. Some of them read files produced by other jobs. Race conditions. Everything was broken in ways that were hard to debug. The fix was obvious in retrospect: stagger the jobs so each one runs after its dependencies are finished.

What you actually need to start

A Python environment. A Claude API key. A way to schedule scripts (cron on Linux, Task Scheduler on Windows, or just a cloud function). A way to receive output, Telegram is free and easy but email works fine.

That's it. No framework. No vector database. No orchestration layer.

I know people who've spent three weeks evaluating LangChain vs. CrewAI vs. AutoGen before writing a single line of code. I understand the impulse. I've done it with other things. But for background cron agents, you don't need any of that. You need a script that calls an API and does something with the result.

The thing that makes it work over time is not the code. It's the context files you feed it. That's where you should spend your energy.

The background I actually needed

I had been writing Python for about two years when I started this. I could read documentation and figure things out, but I wasn't writing clean, well-structured code by any stretch. Most of my scripts were functional and messy.

That was enough. The hard part wasn't the code. The hard part was understanding what context an AI agent actually needs to produce useful output, and how to structure a system so that context accumulates over time instead of resetting every run.

I figured that out by building it badly first, seeing where it failed, and fixing those things. There's no shortcut.

If you're not technical at all, some of this will be a bigger lift. But "not technical" is a spectrum. If you've ever written a Python script or followed a terminal tutorial, you can do this. The actual code is not the bottleneck.
