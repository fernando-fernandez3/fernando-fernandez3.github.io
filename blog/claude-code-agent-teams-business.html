I Don't Use Claude Code Agent Teams for Coding. I Use Them to Run My Business.

Fernando Fernandez 8 min read 1,420 words
By Fernando Fernandez | March 2026

Every article about Claude Code Agent Teams covers the same thing: spawn a frontend agent, a backend agent, and a project manager agent, watch them write code together. That's genuinely cool. But I don't use them that way.

I use agent teams to run a side business while I'm at my day job. Six agents, all on cron schedules, all coordinating through shared markdown files. No coding involved. No interactive sessions. The whole thing runs at 6am without me touching a keyboard.

Here's the setup, what actually works, and what's a bad idea.

What Agent Teams Actually Are (Briefly)

In case you haven't seen it: Claude Code now lets you spawn sub-agents from within a parent agent session. The parent orchestrates, the sub-agents execute, they can pass results back up. It's multi-agent coordination built into the tool.

The default mental model is: one orchestrator breaks down a big coding task, hands off subtasks to specialists, collects results. Good for large codebases. Not the only way to use it.

The key insight that changed how I use this: sub-agents don't have to run interactively. You can configure an agent to spin up on a schedule, do work, write output to a file, and exit. If you chain agents so that one reads another's output, you have a pipeline. That pipeline runs continuously, without you.

That's what I built.

The Six-Agent Setup

My fernandOS Digital system has six agents total, each with a defined job:

1. Researcher - Runs Sunday nights. Scans trending subreddits (r/ClaudeAI, r/LocalLLaMA, r/passive_income), identifies what people are actively asking about, writes a brief to research/weekly-brief.md.

2. Builder - Runs when there's a new brief. Reads research/weekly-brief.md, decides what content to produce, does the actual work (blog posts, product copy, social posts), writes output to the appropriate folder.

3. QA - Runs after Builder finishes. Reads new content, checks it against STYLE.md, flags violations, writes notes to qa/review-notes.md.

4. Reporter - Runs Monday mornings. Reads everything produced in the last week, writes a summary to a Telegram message. This is the only agent that contacts me. Everything else is silent.

5. Memory Consolidator - Runs nightly. Reads daily operation logs, compresses them, updates MEMORY.md, deletes old daily files when they exceed 90 days.

6. Health Checker - Runs every 6 hours. Checks that other agents ran when they were supposed to, pings me on Telegram if anything looks stale.

Coordination between all of them: flat markdown files.

The Coordination Layer Is Just Files

This is the part that trips people up when they first look at multi-agent systems. They expect complex message-passing, shared memory databases, a proper orchestration framework. My system uses none of that.

Each agent writes its output to a predictable path. The next agent in the pipeline reads from that path. That's it.

Researcher writes to research/weekly-brief.md. Builder reads from research/weekly-brief.md. Builder writes to growth/blog/new-post.md. QA reads from growth/blog/new-post.md. QA writes to qa/review-notes.md.

The timing is handled by cron offsets. Researcher runs at 9pm Sunday. Builder runs at 9:30pm Sunday (30-minute gap gives Researcher time to finish). QA runs at 10pm. That sequencing is the "orchestration." Not an orchestrator agent, not a coordinator watching for completion signals. Just staggered cron schedules.

Is this fragile? A little. If Researcher takes longer than 30 minutes, Builder starts on a stale brief. In practice this hasn't been a problem -- Researcher consistently finishes in 8-12 minutes. I set the gap to 30 to have margin.

The more sophisticated version would have Builder check a timestamp in the brief file before starting. I haven't needed to add that yet.

Why Not Use Claude Code's Built-In Agent Teams For This

Honest answer: I tried it. The built-in agent teams feature in Claude Code is designed for interactive sessions. You're present, you're watching, you can intervene. That's fine for coding, where you want to review the output before it lands in your repo.

My background automation doesn't want that. I want it to run at 2am and not need me.

The built-in system also has the orchestrator and sub-agents all sharing the same session lifecycle. When your session ends, everything stops. Cron-based agents don't have that problem -- each one is its own process, starts clean, does its job, exits.

There's also cost. An orchestrator that maintains context across a multi-agent coding session accumulates tokens across all the sub-agents' work. My cron agents each have a tight, bounded context window. Researcher reads about 50KB of subreddit data per run. Builder reads one brief plus the style guide. QA reads one new document plus STYLE.md. None of them are carrying a sprawling 100K-token context.

Daily cost for all six agents: around $0.40. For comparison, a single unconstrained Sonnet session doing comparable work would run $3-6.

What Actually Broke

The first version had four agents instead of six, and no QA step. Builder would produce content directly to the output folder. The content was fine technically but kept violating the style guide in subtle ways. Not egregiously wrong, just not consistently good. I added QA as a dedicated pass specifically because Builder has too much to do in one context window to also perfectly enforce style.

Second thing that broke: I had Builder and Researcher trying to communicate via a shared status.json file with timestamps and flags. Builder would set a flag when it wanted new research, Researcher would check the flag. In theory elegant. In practice, race conditions, stale state, agents checking a flag that got written in the wrong format.

I deleted the status file entirely and went back to cron schedules. Simpler. Less to debug.

Third thing: Health Checker originally ran every hour. Too many pings when things were working fine (it was pinging me with OK statuses I didn't need). Changed it to run every 6 hours and only message me on failure, not success. Now I only hear from it if something's broken.

The Piece Most People Skip: SOUL.md Per Agent

Every agent in my system has its own instruction file. Not a generic system prompt, but an agent-specific file that defines what it's for, what it reads, what it outputs, and how to behave.

Researcher's file tells it exactly which subreddits to scan, how to evaluate signal vs. noise, and what format to use for weekly-brief.md. Builder's file explains the product catalog, links to STYLE.md, and gives it a decision tree for what type of content to produce based on the brief.

The reason this matters: when an agent starts from scratch with a clean context window, it needs to know everything it needs to know in that startup read. It can't ask. It can't remember yesterday's instructions. The agent instruction file is how you give a stateless process persistent identity.

If you skip this and just rely on a generic prompt, your agents will produce inconsistent output. Not wrong necessarily, just inconsistent. They'll make slightly different formatting decisions, slightly different voice choices, slightly different structural choices. Over 100 runs that becomes a mess.

Thirty minutes writing good per-agent instruction files saves hours of debugging inconsistent output later.

Who This Is For

You probably don't need a 6-agent pipeline on day one. The way I'd structure it starting from zero:

Start with one cron agent. Give it one job. A morning briefing, a daily summary, a health check. Get that running reliably. You'll learn where the edge cases are (context window limits, file path issues, cron timing) without debugging six things at once.

Once you have one agent running clean for two weeks, add a second. Probably a memory consolidator -- something that runs nightly and keeps the first agent's memory files from growing unbounded.

Agent three is when you start getting into pipeline territory. Maybe a reporter that reads the first agent's output and sends you a summary. At this point you've essentially built the foundation of what I have.
