My AI Agents Kept Failing in Production. Here Are the 7 Things I Fixed.

Fernando Fernandez 9 min read 1,696 words
~1,800 words | Target keyword: ai agent production failures debugging | SEO priority: High (underserved, high intent, evergreen)

For the first month, my AI agent system was a mess.

Not immediately obvious mess. The kind where everything looks fine, then you check your Telegram at 8am and realize the morning briefing hasn't fired in three days and you have no idea why. Or the agent ran but wrote a response file to the wrong path and nobody noticed. Or it burned through $4 in tokens on one cron job that was supposed to cost $0.12.

I fixed all of it. This is what was wrong and how I solved it.

The Setup, For Context

I run 10+ AI agent cron jobs. They fire on schedules, read from workspace files, do work, write back to those files, and send me Telegram notifications when they're done (or when they break). The whole system costs maybe $1.50–$2.00 a day depending on how much I use it.

The failures I'm describing happened over about six weeks of running this in production. Some were dumb mistakes. Some were subtle architecture problems I wouldn't have predicted. A few were just the API behaving unexpectedly.

Failure 1: Silent Crashes

The first cron job I built would just... stop. No error. No notification. The job would be marked as run in the crontab logs, but nothing would have happened. I'd check the output file and it was from two days ago.

The problem: I was catching exceptions and passing silently. I had a generic except: pass somewhere in my error handling that was swallowing failures. The cron job ran, hit an error, swallowed it, and exited with code 0. Everything downstream thought it succeeded.

The fix was two things. First, I stopped using bare except clauses. Now I catch specific exceptions and log them. Second, every cron job sends a Telegram notification either on success or failure. There's no "exit without telling anyone" path.

The rule now: every job ends with a message. Not a long report. Just something. "Morning briefing done" or "Morning briefing FAILED: check logs." If I don't get a message by 6:15am, I know the cron didn't run at all.

Failure 2: API Timeouts on Large Context Windows

My research agent sends a big batch of notes to Claude and asks it to synthesize them. One day I started getting occasional failures on this job. Not consistent, not every run. Maybe one in ten.

The error was a timeout. The request was taking too long and the client was giving up before the API responded.

I'd been feeding this agent too much context. Research notes from multiple sessions, file contents, background docs. The total was getting up toward 30,000 tokens sometimes. Claude can handle that, but the latency on large prompts is real and variable. On a slow day, it was crossing my client timeout threshold.

Two fixes. First, I trimmed the context. I went through the research agent and started filtering what actually goes into the prompt. Not everything in the workspace folder belongs in every request. Now it only sends what's actually relevant to that day's task.

Second, I increased the timeout on the client from 30 seconds to 90 seconds for that specific agent. For shorter agents I left it at 30. The timeout should match the expected workload.

Failure 3: Writing to Wrong Paths

This one embarrassed me because it was so avoidable.

I had a cron job that was supposed to write a summary to workspace/briefing/today.md. It was writing to briefing/today.md instead, which is a relative path that resolves to wherever the cron job happens to be running from. The cron was running from a different directory than I expected. The file was being created somewhere totally different, overwriting nothing, accumulating orphan files I'd never see.

I found this because I noticed the briefing/ folder in my workspace root had some mystery files.

The fix: use absolute paths everywhere. Not ../briefing/today.md. Not ./briefing/today.md. The full path, /home/user/workspace/briefing/today.md. I went through every cron job and replaced every relative file reference with absolute ones. This is basic but easy to miss when you're building fast.

Also worth noting: test your cron jobs by running them as cron would run them, not as your terminal would run them. Cron has a minimal environment. Your shell aliases, your $PATH, your directory context, none of that is there. I run them with env -i HOME=$HOME PATH=/usr/bin:/bin python3 /path/to/script.py when testing, to catch environment problems before they become production problems.

Failure 4: Memory Files Getting Corrupted

My agents use shared markdown files as their memory. One agent writes, another reads later. It mostly works, but I had a period where my memory files would get weird.

Not corrupted in a technical sense. More like mangled. The agent would append a new summary but include the previous summary twice, or the file would gain random blank lines, or the headings would start drifting. Over a few weeks the files became hard to parse.

The issue: I was telling agents to "update" the memory file without being specific about the format. The agent would read the existing file, decide how to update it, and write back. Reasonable. But Claude interprets "update" loosely. Sometimes it reformatted things. Sometimes it added structure it thought was helpful. Every agent run made the file a little less predictable.

The fix was being more specific in the prompts. Instead of "update the memory file," I now say "append a new section with today's date as the heading. Do not modify any existing content. Use this exact structure: [template]." I also split memory files so different agents write to different sections. Less contention, less drift.

Failure 5: Model Choice Mismatch

I was running everything on Claude Opus because that's what I'd been using in testing and it worked great. Then I looked at my costs and did the math.

My daily standup summarizer was costing about $0.45 per run. Twice a day. That's $27/month for a job that's basically text formatting. It reads bullet points and rewrites them as a short paragraph. That does not need Opus.

I now have a simple rule: Haiku for anything that's formatting, classification, or simple generation. Sonnet for anything that needs judgment, nuance, or multi-step reasoning. Opus for things where quality really matters and I'm willing to pay for it. Most of my cron jobs are Haiku or Sonnet now.

The monthly cost dropped by about 35% with no meaningful drop in quality on the jobs that switched.

Related: watch for model version changes. When Anthropic updates a model version, behavior can shift. I had one job start producing longer outputs after a model update, which pushed it over my max_tokens limit and started truncating responses. I didn't notice for a week. Now I have an expected output length check in my validation step.

Failure 6: Rate Limits During Peak Hours

My system fires several cron jobs at 6am: morning briefing, calendar check, email summary, research queue. All at the same time. For a while this was fine. Then I started hitting rate limit errors during busy API periods.

The jobs weren't that expensive individually. But firing five of them at once with no coordination meant I was occasionally hitting rate limits on concurrent requests.

The fix was staggering the cron schedules. Not all at 6:00. Now it's 6:00, 6:02, 6:05, 6:08, 6:12. Two minutes between jobs. The morning sequence still finishes before I wake up, but the API requests are spread out and I haven't had a rate limit error since.

Small thing. Worth knowing.

Failure 7: Prompts That Worked in Chat, Failed in Cron

This is the subtle one.

I developed most of my agent prompts interactively. I'd test them in the Claude UI or in a notebook, tweak until they worked, then paste them into my cron scripts. Most of the time this worked. But a few prompts behaved differently in production.

The difference: interactive testing has session context. Even when you think you're starting fresh, the framing of a conversation shapes how Claude interprets instructions. "Based on what we've been discussing" doesn't mean anything in a cron job where there is no "we've been discussing."

The other part: I was writing prompts assuming Claude knew things about my setup that I hadn't put in the prompt. I'd type something like "follow the usual format" and Claude in a session context would know what I meant from earlier examples. The cron job had no such context.

I fixed this by treating every prompt as if it's talking to someone who has never met me before. Every prompt now includes: who I am, what I'm trying to accomplish, the exact output format expected, and at least one example. It feels redundant when writing it. The cron jobs got noticeably better.

The Pattern

Looking at these seven failures, most of them share a root cause: I built the system fast and didn't account for the difference between "works once" and "works reliably at 6am every day without anyone watching."

Interactive development is forgiving. You're there to notice when something goes wrong. Production is not forgiving. Nobody's watching. The job fires, does whatever it does, and either your Telegram notification comes through or it doesn't.

The discipline that made everything more stable was treating every cron job like a small piece of infrastructure. Clear inputs. Clear outputs. Explicit error handling. Observable behavior (the Telegram notifications). No silent exits. Absolute paths. Specific prompts.
