What My 10 AI Cron Jobs Actually Do for Me Every Day

Fernando Fernandez 7 min read 1,264 words
By Fernando Fernandez · Staff Support Engineer at Tesla, runs a production AI agent system on a personal Linux server

I run 10 AI cron jobs a day. I've been doing this for months. A few of them changed how I work. Most of them are invisible in the best possible way. One of them turned out to be useless and I killed it.

This is not a tutorial. I have [another post for that][link-cron]. This is about what the system actually does to your daily life when it's running.

The 6am briefing

The most valuable one runs at 6am.

My agent pulls my calendar, checks my email, reviews any open tasks in my notes, and sends me a Telegram message before I'm out of bed. Short. Four or five bullet points. What's on today, anything urgent that came in overnight, one thing to focus on.

I check it while I'm still in bed. By the time I'm making coffee, I already have a map of the day.

Before I had this, I'd check email and Slack the moment I woke up and spend the first 20 minutes of my morning reacting to whatever was loudest. The briefing changed that. I'm reading a summary, not a firehose. The agent has already decided what matters.

That said: it took three or four iterations to get the prompt right. The first version was too long. The second version missed things I actually cared about. The current version is tuned to my workflow and it's good enough that I notice when it doesn't run.

Memory maintenance (the one you don't notice)

Every night at 11pm, a job runs that compresses my memory files.

My agent uses a three-file memory system: a daily log, a user context file, and a project/agent instructions file. Over time, the daily log gets long. The maintenance job summarizes older entries and trims it down.

I never think about this. I just notice that my agent's context never goes stale and I never have to manually manage any files.

The job uses Haiku (the fastest, cheapest Claude model). The whole thing costs a few cents a night.

JIRA queue review

I'm a Staff Support Engineer at Tesla. My team gets a steady flow of JIRA tickets from stakeholders across the company. Feature requests, issue reports, questions, things that probably belong in a different queue.

I have a cron that runs every weekday morning and reviews the open queue. It flags tickets that are stale (older than three days with no update), surfaces anything with language that suggests urgency, and notes tickets that are missing information I'd need to actually work them.

This saves me probably 20 minutes of triage a day. That's not a lot. But it's the cognitive overhead of triage that was the problem, not the time. I'd start the morning with 30 tickets to mentally sort through. Now I start with a sorted list and a short note explaining the sorting logic.

The agent is wrong sometimes. It flags things as urgent that aren't, misses context I have about certain stakeholders, underestimates priority on certain ticket types. I still do the final call. I just don't start from zero.

Spanish practice

This one is different from the others.

I wanted to practice Spanish consistently but I don't have a formal schedule and I kept skipping apps like Duolingo. So I set up a cron that sends me a voice note in Spanish every day at noon. Five to seven minutes. Conversational, not classroom. The topic rotates: vocabulary, real-world scenarios, current events.

The voice is ElevenLabs, native Spanish accent. My phone gets a voice message from what sounds like a person. My brain processes it differently than text.

I've been more consistent with Spanish practice since I set this up than I was in the two years before. That's a real result. I'm not going to oversell it (I'm still far from fluent), but I was barely practicing before and now I practice every day.

This is the one my family asks about when I explain the system. It's the most concrete example of the agent doing something a human would never do for you (send you a five-minute personalized language lesson every single day).

The one I killed

I had a cron that wrote a daily summary of everything my agent did that day. Timestamped log, what jobs ran, what they produced, what errors came up.

In theory: great. In practice: I never read it.

I thought I'd want the audit trail. I thought I'd review it weekly and tweak things. I don't. When something breaks, I notice immediately because the output stops coming. When everything works, I don't need a summary to tell me it worked.

The job was using maybe 30 cents a day and producing a file I never opened. Killed it after six weeks.

The lesson: don't build crons you think you'll want. Build the outputs you actually check.

Health check

Once a day, a small job checks that all my other cron jobs are still running. It writes a timestamp when it runs. A second job checks that timestamp and alerts me via Telegram if it's stale.

I've gotten two alerts in several months. Once when I ran out of API credits (forgot to top up). Once when my server rebooted and cron didn't restart cleanly.

Both times I had the alert before I noticed anything was wrong, which makes all the other jobs trustworthy.

If you set up a cron system and don't set up monitoring, you'll eventually have a cron that silently failed and you won't know for days. Build the health check first.

What this costs

About $3 a day. Sometimes $2.50, sometimes $3.50. Steady.

The briefing and the Spanish practice use Sonnet. The maintenance job and health checks use Haiku. Interactive sessions throughout the day use whatever model fits the task.

I'm on Anthropic's Max plan ($100/month). The cron jobs cost about $90 of that. The rest is interactive use.

Before I had model routing, I was running everything on Sonnet. That would have cost about $270 a month for the same workload. Model selection is the single biggest cost lever.

What makes a good cron job candidate

The jobs that work share a few things:
- Clear, bounded task. The agent knows when it's done.
- Output I'll actually look at. Briefing, message, file I open (not a log I ignore).
- Doesn't need my input mid-task. If I have to clarify something, it can't run while I sleep.
- Failure is visible. If the briefing doesn't show up, I notice. If the Spanish lesson doesn't arrive, I notice.

The jobs that fail tend to be too open-ended, produce output I don't check, or require context only I have.
