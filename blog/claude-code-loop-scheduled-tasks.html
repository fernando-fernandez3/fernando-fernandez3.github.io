Claude Code /loop Shipped Last Week. The 3-Day Cap Is the Point.

Fernando Fernandez 6 min read March 10, 2026
Claude Code now schedules recurring tasks inside your session. People are calling the 3-day limit artificial. I think it's the most honest thing about the feature.

Claude Code shipped /loop last Friday and people are excited. Reasonably so. You can now schedule recurring tasks inside a Claude Code session — check error logs every few hours, auto-create PRs for fixable bugs, generate a Slack summary at end of day. Standard cron expressions, natural language reminders, up to 50 concurrent tasks per session.

I've been running the equivalent of this for 8 months on direct API + system cron. 10 background agents, daily execution, about $40/month total. So I've thought about this design space a lot.

One comment on Reddit called the 3-day auto-delete cap an "artificial ceiling." I don't think so. I think it's the most informative detail in the whole announcement.

What /loop Actually Does

Quick overview for anyone who missed it. Inside a Claude Code session, you can now run something like:

/loop every 2h check the error logs in ./logs/ and open a PR if you find anything fixable

It schedules that as a recurring task at whatever interval you specify — minutes, hours, or days. Up to 50 concurrent tasks per session. Natural language scheduling also works: "remind me at 3 PM to push the release branch." Uses your local time zone. Standard cron syntax if you prefer that.

The key phrase in Anthropic's announcement: "as long as Claude Code is active."

This runs in your current session, on your machine, while Claude Code is open. The moment you close it, the jobs stop. Sessions auto-delete after 3 days.

That's not a complaint. That's the design.

The 3-Day Cap Is Not a Limitation

Every article about /loop has treated the expiration as something Anthropic will eventually remove. "Artificial ceiling," people keep saying. I disagree.

3 days is a perfect window for dev-time automation. You're working on a sprint, a PR review cycle, a debug session. You want something running in the background while you're heads-down. Then the task is done, the context is stale, and the job should stop anyway.

That's different from what I run on system cron. My morning briefing has been running at 6:30 AM every single day for 8 months. My memory consolidator runs at 11 PM nightly. Those aren't tied to a coding session. They're infrastructure. They run whether I'm at my computer or not, whether Claude Code is open or not.

/loop is session automation. System cron is infrastructure automation. They're different things solving different problems, and treating /loop's 3-day cap as a deficiency misreads what the feature is for.

Where /loop Genuinely Wins

There are real cases where /loop is better than what I'm doing.

Anything that needs live codebase context. My system cron agents hit the API with pre-loaded markdown files. They can't browse actual code files in real-time the way a Claude Code session can. If you want an agent to check error logs and fix actual bugs — not just summarize them, actually open a PR with a fix — /loop has access to your project tree in a way a standalone API call doesn't.

PR monitoring is the obvious case. Anthropic's own examples focus here: check for CI failures, create fix PRs, self-review. That work requires knowing the codebase in detail. /loop has that context natively. My API cron agents would need me to pre-load relevant files every time, which defeats the point.

Zero setup is also genuinely nice. No server to maintain, no API keys to manage separately, no crontab entry. You're already in Claude Code. One command and the task is scheduled.

The "remind me at 3 PM" natural language scheduling is something I didn't know I wanted until I read about it. For dev-time reminders — push the release branch, run the benchmarks, check the deploy — that's a legitimate quality of life improvement.

Where System Cron + Direct API Wins

Headless execution is the real one. My agents run at 6:30 AM while I'm asleep. My laptop doesn't have to be on. Claude Code doesn't have to be open. The job runs, sends me a Telegram message, and I read it when I wake up.

/loop can't do that. If your machine is off or Claude Code is closed, the job doesn't run.

Duration is the other constraint. I have agents that have run every day for 8 months. Nothing about /loop supports that. If you wanted truly long-running automation, you'd have to rebuild the session repeatedly, which is just system cron with extra steps.

Non-coding tasks are where system cron shines. Most of my agents aren't doing development work. Morning briefing. Task extraction from inbox. Email digest. Memory consolidation. Research summarization. None of those need codebase context. System cron handles them fine, and for cheaper, because I route them to Haiku.

Cost control matters when you're running 10 agents daily. With direct API you pick the model per task. My health checker runs on Haiku — $0.002 per run. My memory consolidator runs on Haiku too. Only the tasks requiring real judgment go to Sonnet. /loop runs inside a Claude Code session, which is Sonnet-class across the board. For tasks where Haiku output is indistinguishable, you're paying 15x more per run than you need to.

Model flexibility. My cron setup isn't locked to Claude. If I need to route a specific task to GPT-5.4 or Gemini for cost or capability reasons, I can. /loop is always Claude Code.

The Actual Comparison

Setup Zero — already in Claude Code Minutes to configure
Codebase context Full (live Claude Code session) Pre-loaded markdown files only
Headless execution No — requires active session Yes — runs on server, no GUI needed

Best task types Dev-time: PR review, bug fixes, CI Background: briefings, digests, notifications

Natural language scheduling Yes — "remind me at 3 PM" No — cron syntax only

Who Should Use Which

Use /loop if:

  • You're actively in a Claude Code session and want background tasks during that session
  • The task genuinely needs codebase access — PR monitoring, log analysis with auto-fix, CI watching
  • You don't want to set up a server or manage separate API credentials
  • Under 3 days is fine for your use case

Use system cron + direct API if:

  • You need overnight or long-running automation — things that should run whether you're at your computer or not
  • You're building non-coding agents: briefings, task extraction, email digest, memory consolidation, notifications
  • Cost optimization matters — routing Haiku-tier tasks to Haiku saves real money at scale
  • You want model flexibility across providers
  • You want to actually own the infrastructure, not have it tied to a UI session

The honest answer for most people: start with /loop to see what's possible. If you hit the 3-day limit or find yourself wishing it would run while your laptop is closed, that's the moment to look at the direct API path.

The /loop launch and the Cursor Automations launch and the Anthropic Marketplace all point to the same thing: everyone is building session-layer automation. The headless, always-on layer is still mostly DIY. That's either a gap or an opportunity depending on what you're trying to build.
