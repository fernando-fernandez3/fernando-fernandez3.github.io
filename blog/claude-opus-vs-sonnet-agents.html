When Should You Upgrade to Opus? A Real Decision Guide for Agent Builders.

Fernando Fernandez 8 min read March 10, 2026
10 agents running in production. Here's what each one actually runs on, and why — with a decision matrix for every agent type.

The question I get asked more than any other in the r/ClaudeAI comments: "Should I be using Opus instead of Sonnet?"

The honest answer: probably not, for most of your agents. But there are specific tasks where Sonnet will fail you in ways that cost real time to debug, and Opus just handles them. The trick is knowing which tasks those are before you waste two weeks on the wrong model.

This isn't about benchmarks. SWE-bench scores don't tell you what happens when your agent runs at 2am and has to synthesize contradictory search results into a coherent briefing. Real task behavior does.

Here's how I think about it after running 10 production agents for months.

The Mistake Most People Make

The default move is Sonnet for everything. I did this. Most people do. Sonnet is the safe middle option — good output, reasonable cost, doesn't embarrass you.

The problem is that "safe" doesn't mean "right." It means you're paying Sonnet prices for tasks that Haiku handles identically, and you're running into subtle quality failures on tasks that actually needed Opus.

Both of those cost money. The first costs you literally — you're overpaying per run. The second costs you in time — you're staring at output wondering why your summarization agent keeps burying the lead.

Model routing isn't complicated. It's just asking, per task, "what does this actually require?"

Tier Breakdown: What Each Model Actually Does Differently

Before the decision guide, a quick reality check on what the tiers are.

As of March 2026, the relevant models and rough API pricing:

  • Haiku — ~$0.25/$1.25 per million tokens (input/output). Fastest, cheapest.
  • Sonnet — ~$3/$15 per million tokens. The workhorse.
  • Opus — ~$15/$75 per million tokens. 5x Sonnet on output. Slow.

The performance gap between tiers isn't uniform. It depends on task type. Haiku closes the gap significantly on structured tasks. Opus separates significantly on open-ended synthesis where quality variance is expensive.

Category 1: Haiku Is Enough

These are tasks with clear inputs, defined output structure, and no requirement for judgment.

Good fits:

  • Health checks (read a file, check a value, send an alert)
  • Simple reformatting (take a JSON blob, output a formatted Telegram message)
  • Classification with tight categories (is this email urgent/not urgent/spam)
  • Data extraction from structured sources (parse a CSV, pull three fields, write to SQLite)
  • Short summarization where the source is already clean

I run four of my ten agents on Haiku. The health checker, the email screener classifier, the weather formatter, and the daily task extractor. None of them have had a quality regression from being on Haiku. They just run faster and cost less.

Signs you've put the wrong task on Haiku:

  • Output occasionally drops required fields without telling you
  • Classification accuracy feels slightly off but hard to pinpoint
  • Long documents get shallow treatment that misses the actual point

If you're seeing those, move up to Sonnet. Don't move to Opus yet.

Category 2: Sonnet Is the Right Call

Sonnet is where most actual agent work lives. Tasks that require following multi-step instructions reliably, producing consistent output quality across varied inputs, and handling moderate ambiguity without falling apart.

Good fits:

  • Writing that will actually be read (briefings, summaries, draft emails)
  • Multi-step pipelines with several dependent tasks
  • Classification where the categories have nuance
  • Code review for straightforward issues
  • Research synthesis from 2-3 sources where they mostly agree
  • Following complex instruction sets reliably

Six of my ten agents run on Sonnet. The morning briefing agent, the research agent, the QA reviewer, the social monitoring agent, the weekly summary writer, and the memory consolidator. For all of them, Sonnet produces output I'd want to read and takes instructions without drift.

The briefing agent is the important one to note. It runs daily and I actually consume its output. That's a real quality bar. Sonnet clears it consistently. I tested Haiku on it briefly — the briefings were shallower, missed context that mattered, and occasionally restructured information in ways that buried the important stuff. Back to Sonnet.

Category 3: When Opus Earns Its Cost

Opus earns its cost in a narrow but real category: tasks where quality variance is expensive and the task genuinely requires holding multiple conflicting pieces of information while making a judgment call.

Good fits:

  • Deep research synthesis where sources disagree or are nuanced
  • Long-document analysis where catching the right signals matters
  • Complex code architecture decisions, not just review
  • Tasks where you need the agent to push back or flag problems, not just comply
  • Multi-turn reasoning chains where earlier mistakes compound

The critical filter: would a quality miss cost you significant time to catch and fix? If yes, Opus might be worth it. If the output gets reviewed anyway before it matters, Sonnet is probably fine.

I run zero agents on Opus in my regular cron setup. The one place I've used it is on-demand research tasks — when I'm asking a one-off question that requires synthesizing 10+ sources and I need a real answer, not a competent summary. For automated cron, Sonnet handles everything I've thrown at it.

But I work in tech support engineering, not investment research or legal analysis. If your agents are doing work where quality errors have real consequences, Opus becomes more defensible.

The workflow: start on Sonnet, try downgrading to Haiku, only upgrade to Opus if you're seeing specific symptoms and quality matters enough to justify 5x cost.

Don't start on Opus trying to optimize later. You'll never downgrade. I've watched too many people in the comments run their entire setup on Opus because they were worried about quality, spend way too much, and never benchmark whether Sonnet would have been fine.

What Happens When You Get It Wrong

Under-tier (Haiku when Sonnet was needed): subtle quality degradation that's hard to attribute. Your briefing feels slightly off. Your classification is almost right. You spend time wondering if your prompts are bad when the model is the issue.

Over-tier (Opus when Sonnet was needed): you pay 5x more and get the same output you'd have gotten from Sonnet. Opus doesn't make simple tasks better. It makes hard tasks more reliable. Simple tasks are already reliable on Sonnet.

The diagnostic move: if you're not sure whether a task needs Sonnet or Opus, run the same prompt on both with the same input 5 times. Look at the variance in output. If Sonnet's variance is acceptable, stay on Sonnet. If Opus consistently produces noticeably better judgment calls, it might be worth it.

The Migration Pattern That Works

When you're setting up a new agent, this is the sequence that wastes the least money:

  1. Start on Sonnet
  2. After a week of real runs, review the outputs critically
  3. If outputs are clearly meeting your bar: test Haiku on 20 runs. Compare. If Haiku is fine, switch.
  4. If outputs are NOT meeting your bar: identify whether it's a model issue or a prompt issue. Fix the prompt first. If the prompt is solid and the model keeps missing, test Opus on 20 runs. If Opus fixes it, decide if the cost is worth it.

The "fix the prompt first" step matters. A lot of what people think is a model capability gap is actually a prompt issue. Opus will sometimes power through a bad prompt and give you a decent answer. Sonnet and Haiku will fail more visibly. That's actually useful feedback. Don't chase model upgrades as a substitute for prompt work.

The Short Version

Default to Sonnet. Test Haiku on everything after the first week. Only go to Opus for tasks where quality variance is genuinely expensive and you've verified that's a model issue, not a prompt issue.

Most builders I talk to use tier 1 and tier 2. Almost nobody needs tier 3 in their automated cron setup.
