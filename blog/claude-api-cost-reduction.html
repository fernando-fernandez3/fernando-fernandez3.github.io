How I Cut My Claude API Costs by 60% With Model Routing

Fernando Fernandez 6 min read 1,079 words
By Fernando Fernandez · Staff Support Engineer at Tesla, runs 10+ AI agents daily

The first month I ran my agent setup seriously, I looked at my API usage and did the math wrong in the best possible way. I thought: "this should cost less." Then I looked at why it didn't, fixed one thing, and watched the cost drop.

That one thing was model routing.

The Problem With the Default

Most people getting started with Claude default to Sonnet for everything. It's the middle option. It feels safe. It produces good output. Why wouldn't you use it?

Because tasks aren't all the same complexity, and model pricing reflects that.

Haiku is significantly cheaper than Sonnet. Sonnet is significantly cheaper than Opus. The performance gap between them depends on the task. For a health check that reads a timestamp and sends a Telegram message if a threshold is exceeded, Haiku does that as well as Sonnet. For writing a detailed technical summary that I'll actually read, Sonnet earns its cost. For a task that genuinely requires complex multi-step reasoning, Opus.

When you run everything on Sonnet, you're paying Sonnet prices for tasks that only need Haiku.

What Model Routing Actually Means

Model routing is deciding, per task, which model to use. Not "what's the best model I have access to." The minimum model that does the job well.

It's a habit change more than a technical change. The technical part is just a field in your cron config or prompt setup: model: claude-haiku-4-5 vs model: claude-sonnet-4-5.

The habit part is asking, before you automate anything: "what does this task actually require?"

The Decision Tree

I run 10 cron jobs. Here's how I categorize them:

Use Haiku when:
- The task is a read/check/conditional (health checks, heartbeat monitoring)
- The task is summarize/extract/compress with no quality bar on the prose (memory maintenance, daily note extraction)
- The output is structured data, not something you'll read (token reports, log parsing)
- The task is templated (vocabulary quizzes, simple formatted notifications)

Use Sonnet when:
- The output is something you'll actually read (morning briefings, daily summaries)
- The task requires judgment, not just pattern-matching (ticket review, research synthesis)
- Quality matters and the cost delta is worth it

Use Opus when:
- You genuinely need the best reasoning available
- The task is complex enough that Sonnet is noticeably worse
- This is rare for scheduled tasks (roughly 5% of my calls)

My split as of early 2026: ~60% Haiku, ~35% Sonnet, ~5% Opus.

Real Before and After

I didn't track my costs carefully at the start, which was its own mistake. But I can work backward from what I know.

Before model routing, I was running most jobs on Sonnet. The Spanish practice crons, the health check, the memory maintenance, all on Sonnet because that's what I defaulted to.

After routing: those jobs moved to Haiku. That's 4-5 of my 10 jobs. Each one is now 4-5x cheaper per run.

Estimated savings on cron jobs alone: 50-60%. My total daily cost including interactive sessions is around $3. Before routing it was closer to $7-8 for the same workload.

The Other Lever: Context Trimming

Model routing is the big one. But what you load into context multiplies the cost too.

Every file your agent reads is tokens. Every previous message in the conversation is tokens. If you're loading your full memory archive into every session, you're paying for all of it even if most of it isn't relevant.

What I do instead:

  • My agents load MEMORY.md (the summary file, kept under 10KB by nightly compression) plus the relevant daily note for today and yesterday.
  • If they need to look something up in older files, they know where the archive lives and can search it.
  • They don't load everything by default.

This is context management by design. You decide what the agent reads, not "everything available."

The memory maintenance cron exists specifically to keep MEMORY.md from growing unbounded. It runs every night, compresses what's no longer active into the archive, keeps the summary current. That's a Haiku job because compression is a cheap read/write task.

The Token Dashboard

You can't optimize what you can't see. Before I built the token dashboard, I was guessing at where my costs went.

The dashboard is a simple HTML file that reads my usage logs and visualizes cost by model and by job type. It updates daily from the nightly token report cron.

What it showed me that I couldn't have guessed: the JIRA queue review was my second most expensive job, not my first. The morning briefing was first (longer output, Sonnet), but the ticket review was running more often than I thought and with more context loading than necessary.

Trimmed the context on the ticket review. Moved some parts of it to a two-pass approach (Haiku for initial categorization, Sonnet only for tickets that need a detailed review). Cost of that job dropped significantly.

The Compounding Effect

Model routing is a one-time decision that keeps paying off. Every cron job you add, you make this decision at setup time instead of optimizing it later.

The discipline carries over to interactive sessions too. If I'm asking my agent to do a quick lookup or run a check, I pick Haiku. If I'm working through something complex, Sonnet. I don't default to the heaviest model anymore.

At $3/day, my agent setup costs about the same as a coffee. For what it does (10+ automated jobs, morning briefings, language practice, family coordination, JIRA review), that's not expensive. It's just that before model routing, it cost twice as much for the same output.

Summary

The short version:

  1. Default Sonnet for everything is the most common mistake.
  2. Match model to task complexity, not "what's best."
  3. Haiku for reads/checks/maintenance. Sonnet for generation. Opus rarely.
  4. Trim context. Don't load the full archive when you only need the summary.
  5. Build a token dashboard so you can see where costs go.
