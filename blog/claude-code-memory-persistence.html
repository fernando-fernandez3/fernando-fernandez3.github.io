How to Give Your AI Agent Memory That Persists Across Sessions

Fernando Fernandez 8 min read 1,423 words
By Fernando Fernandez · Staff Support Engineer at Tesla, runs Claude Code daily for personal and work projects

Every Claude Code session starts blank. The API is stateless by design, with no memory of previous conversations unless you build it yourself.

That's fine. I'm not complaining about it. What I am saying is that if you want an agent that knows your name, your projects, and what you were working on last Tuesday, you have to build that layer. It doesn't come out of the box.

This post is about how I built it, what I tried that didn't work, and the setup I actually run.

The Problem in Practice

Before I had a memory system, a typical Claude Code session started like this:

I'd open the terminal, start a session, and spend the first few exchanges re-explaining context. "I'm a staff engineer at Tesla. I work in Python. My current project is a pool automation system that runs on Home Assistant. Last time we worked on the ESPHome config for the chlorine sensor, and we ran into a problem with..."

Ten minutes of context-setting before I could get to the actual question.

And that was just for projects I remembered to explain. Anything I'd figured out two weeks ago and hadn't written down was gone. The agent had no idea we'd solved that problem. We'd solve it again.

Multiply that by daily use and it gets expensive in two ways: time and tokens.

Three Approaches I've Seen People Try

1. Memory plugins and integrations. There are tools that bolt memory onto Claude. I've looked at them. The problem is they're platform-dependent, they add another thing to maintain, and they handle the data in ways you don't fully control. Not interested in that.

2. Long chat history / context stuffing. Paste the last five sessions into the context window. This works until it doesn't. The context window has a limit, history fills it fast, and half of what's in there isn't relevant anyway. It's also not searchable. You end up with a giant blob of text and no good way to find specific things in it.

3. File-based memory. This is what I use. The agent reads and writes plain text files. Simple, versionable, fast, and completely under my control.

The file-based approach is what the rest of this post covers.

The Three-File System

My memory architecture has three files. Each one does a different job.

MEMORY.md is the long-term store. It holds things that matter across multiple days: decisions I've made, projects I'm working on, preferences, useful context that would take too long to re-establish. The agent reads this at the start of every session. It writes to it at the end of sessions where something worth keeping happened.

The format is loose. Dated sections at the top for recent stuff, evergreen sections below for things that don't change much. Project notes, important decisions, reference info I look up regularly.

USER.md is my profile. Who I am, what I work on, what I care about. It doesn't change unless I change it. The agent reads it at startup, not to personalize responses in a creepy way, but because knowing the user context upfront means I don't have to re-explain it.

Example of what's in mine: my job, my timezone, my preferred tools, my current active projects, things I'm interested in. Two pages, tops. The agent reads this once and it's loaded for the session.

AGENTS.md is the instruction file for the agent itself. How it should behave, what it should do at session start (read the other files), what its responsibilities are, how to handle edge cases. Each agent I run has its own AGENTS.md. My personal assistant agent has different instructions than my coding agent.

Three files. That's the whole system at its core.

How the Agent Uses Them

At session start, the agent reads all three files. This happens automatically because AGENTS.md tells it to.

During the session, the agent operates with that context loaded. It knows the projects, the history, the preferences. If something comes up that it should remember for next time, it notes it.

At session end (or when something significant happens), the agent writes to MEMORY.md. A brief dated entry: what we worked on, what was decided, anything that would help future-me or future-agent understand what happened.

Over time, MEMORY.md grows. That's expected. The daily notes are in memory/YYYY-MM-DD.md files, and MEMORY.md is the summarized, curated version that the agent actually reads each session. The daily notes are the raw archive.

How to Set It Up

Step 1: Create the directory structure.

workspace/
  MEMORY.md
  USER.md
  AGENTS.md
  memory/
    2026-03-01.md
    2026-03-02.md
    ...

Step 2: Write USER.md now.

Don't skip this. Spend 15 minutes writing out who you are, what you work on, what tools you use, what you're currently working on. This is the context you'd give someone if they were going to help you with your work for the next year.

It doesn't have to be long. Mine is about 40 lines. The point is that the agent has this baseline without you having to explain it.

Step 3: Write your AGENTS.md.

The key instruction is: at the start of every session, read MEMORY.md, USER.md, and today's daily note (plus yesterday's if early in the day). After every meaningful session, write a brief entry to the daily note.

The rest of AGENTS.md is about behavior: how you want the agent to communicate, what it should do vs. ask about, any standing preferences. This varies by agent type.

Step 4: Start MEMORY.md with something useful.

Don't leave it blank. Seed it with the context that would be most valuable for an agent to know. Your current projects, recent decisions, anything you'd want to explain to a new assistant on day one.

Step 5: Let it grow.

The agent writes to it. You occasionally review it and prune what's outdated. Once a week, spend five minutes cleaning up entries that no longer matter. That's it.

The Memory Maintenance Cron

There's one problem with file-based memory: it grows without bound. If you don't manage it, MEMORY.md eventually becomes the size of a novel and takes a while to load.

The fix is a nightly memory maintenance cron. A scheduled task that runs every night, reads the daily notes from the last two weeks, compresses anything older than that into an archive, and keeps MEMORY.md under a target size (I use 10KB as a soft limit).

This runs on Haiku, because compression is a simple read/write task. No generation required, just reorganization.

The maintenance cron does three things:

  1. Compresses daily notes older than 14 days into memory/archive/YYYY-MM.md files (monthly archives)
  2. Trims MEMORY.md if it's grown beyond target size, preserving the most recent and most referenced entries
  3. Writes a brief report confirming it ran and what it compressed

After running this for a month, the system is self-managing. Memory accumulates during active use, maintenance runs overnight, the summary file stays readable.

Why This Works Better Than the Alternatives

The file-based system has some properties that matter over time.

It's versionable. If you're in a git repo, your memory files are in git. You can see exactly what the agent wrote, when, and why. If it writes something wrong, you can fix it.

It's searchable. Plain text files are trivially searchable. grep works on them. If you want to know what you decided about X three months ago, you can find it.

It's portable. These files work with any model, any tool, any interface. They're not locked to a specific plugin or platform. If you switch tools, the memory comes with you.

It's auditable. The agent's memory of you is right there in a file you can read. Not locked in a black box you have to trust.

The main downside: you have to set it up yourself. That's also the point. You control it.
