I Built a 6-Agent System That Runs My Side Business While I Sleep

Fernando Fernandez 9 min read 1,613 words
By Fernando Fernandez · Staff Support Engineer at Tesla, runs a production AI agent system on a personal Linux server

I have a cron job that runs every 30 minutes. It wakes up a Python script, checks in with six AI agents, and logs what they're working on. I don't usually see any of this. It just happens.

By the time I checked in last Sunday morning, those agents had written three full product guides, ten SEO blog posts, a four-email welcome sequence, three launch playbooks, and a set of social media posts for three separate product launches. I was asleep for most of it.

This is not a flex. The setup is maybe 200 lines of Python and a handful of markdown files. But it does illustrate something about multi-agent systems that I think most tutorials skip entirely: the hard part isn't the agents. The hard part is coordination.

Why one agent isn't enough

My first agent was a single Claude instance. It got a task, did the task, reported back. That worked fine for simple stuff: daily briefings, email summaries, reminders.

When I started building a content business on the side, that model broke down fast.

The problem is that "build a content business" isn't a task. It's a project with dozens of interdependent tasks across different domains: product development, copywriting, market research, SEO, launch planning, quality review. A single agent can do any of those things. But giving it all of them at once is like hiring one person to simultaneously be your product manager, your writer, your researcher, and your marketing team.

They'll get confused. They'll lose track of context. The output gets worse as the scope gets bigger.

So I split the work.

The six agents

Here's what I actually built:

The Orchestrator isn't an agent in the Claude sense. It's a Python script that runs on a cron. Every 30 minutes, it wakes up and decides which agents need to run. It reads a shared state file, checks for open issues, and sends heartbeats to the right agents in sequence. Nothing runs without the orchestrator saying so.

The Builder is me writing this post right now. Its job is the actual production work: writing guides, building HTML landing pages, writing code, reviewing assets. When the ProductLead says a guide needs to exist, the Builder writes it.

The Researcher evaluates new product ideas. When I want to know if there's a market for something, I add it to the research queue in the roadmap. The Researcher studies the competition, estimates the audience, and writes a brief. I've killed three product ideas based on its research. That saved me a lot of time.

The ProductLead manages the active products. It defines specs, tracks what's built versus what's missing, and updates the roadmap with current status. When I look at the roadmap and understand exactly where each product stands, that's the ProductLead keeping it current.

The GrowthLead handles distribution. Blog posts, social media copy, email sequences, launch playbooks. The actual words that go in front of potential customers.

The Reviewer does quality control. It reads drafts and checks them against the style guide. It catches AI-isms, bland phrasing, generic structure. When the Builder or GrowthLead writes something, the Reviewer has standing orders to call out anything that sounds like it came from a chatbot.

The Strategist is the long-range thinker. It looks at the overall portfolio, identifies gaps, and makes recommendations. I talk to it when I'm deciding what to build next.

How they actually communicate

This is the part that most multi-agent writeups skip.

Every agent reads and writes to the same set of markdown files. There's a ROADMAP.md that describes what's been done, what's in progress, and what's next. There's a STYLE.md that defines the voice every piece of writing should follow. There are product directories with specs, drafts, bundle assets.

The agents don't talk to each other directly. They talk through files.

When the ProductLead updates the roadmap with a new task, the Builder reads that update on its next heartbeat and picks up the work. When the Reviewer flags a style violation, it edits the file with the fix. When the Researcher kills a product idea, it logs the reason in the research queue so no future agent wastes time on the same idea.

This sounds slow. It's not. The orchestrator runs every 30 minutes. Each agent that gets woken up reads the current state, does whatever work it can, and updates the shared files. By the end of a day, the cumulative output is significant.

The actual results

In one session last weekend, the six agents produced:

  • A 2,785-word product guide (Ship Your AI Agent, Product #1)
  • A 4,500-word pool automation guide (Product #2)
  • A full voice AI guide across 6 chapters (Product #3)
  • Landing pages for all three products
  • Gumroad listing copy for all three
  • 10 SEO blog posts covering the main search terms in this niche
  • A 4-email welcome sequence
  • A styled printable cheatsheet as a lead magnet
  • HTML infrastructure for the blog with a generator script
  • Three separate launch playbooks with step-by-step instructions

The quality is not perfect. The guides have placeholder sections that need my personal stories, real measurements, specific numbers only I know. The agents flagged those sections themselves. They knew what they could write and what they couldn't.

That's something I didn't expect: the agents are pretty good at knowing the limits of what they can produce without my input. The ones that required real experience from my life came back with [FILL IN: your actual cost numbers here] instead of making something up.

What broke along the way

A few things didn't work at first.

Context bleed. Early on, I had the agents share too much context. Every agent read every file. The Researcher was accidentally referencing the ProductLead's spec docs when they weren't relevant. The fix was scope limiting each agent's read access. The Builder reads product specs and style guides. The Researcher reads the research queue and niche briefs. They don't need each other's working files.

Conflicting updates. Two agents trying to write to the same file at the same time caused issues a couple of times. The orchestrator now staggers agent wake-ups by 5 seconds and runs them in a fixed sequence. Not elegant, but it works. I haven't had a conflict since.

Drift from the style guide. The first few drafts had em dashes everywhere. Long inspirational sentences at the end of sections. The phrase "Here's the thing" appearing multiple times per document. I had to make the Reviewer aggressive about catching violations, and I had to make the style guide extremely specific about what "wrong" looks like, not just what "right" looks like.

The Strategist going off-scope. This one is more philosophical. A long-range planning agent, if you let it, will keep generating new work forever. It found a product opportunity in every topic it touched. I had to give it a hard cap: evaluate no more than two new product ideas per week, and only add them to the backlog if the Researcher gives a GO verdict first. Unconstrained scope is how you get a beautiful system full of half-finished things.

What you actually need to build this

Less than you'd think.

The orchestrator is a Python script. It reads a config file with each agent's role description and wake conditions. It calls the Anthropic API with the appropriate system prompt and the agent's relevant context files. It logs output. That's it.

Each agent is a system prompt plus a set of files to read. The system prompt defines the role, the scope, the rules. The files provide the current state of the project. When the agent runs, it reads those files, does work, and writes updates back.

The communication protocol is markdown. No message queues, no databases, no vector stores. A flat directory of files that every agent can read and write. It's boring infrastructure. It works.

If you've already built a single production agent (memory files, cron triggers, Telegram notifications), the jump to multi-agent is mostly about adding structure: clear role separation, a shared state file, and an orchestrator that decides who runs when.

Why this matters

I'm not arguing that everyone needs six AI agents running their side projects. Most people don't.

But the pattern is useful even at smaller scale. The moment you have more than one type of work, you benefit from role separation. An agent that writes is better at writing when it's not also trying to manage a project. An agent that reviews is better at reviewing when it's not also trying to generate content.

The actual bottleneck in this system is me. The agents can produce as much as I give them clear direction on. When I write a specific task in the roadmap, something gets built. When the spec is vague, I get something vague back. That's not a failure of the system. That's just how work works.

The agents are fast and tireless. I still have to decide what to build.
