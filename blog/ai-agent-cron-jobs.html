How to Run AI Agents on Cron Jobs (10 Real Examples With Costs)

Fernando Fernandez 9 min read 1,690 words
By Fernando Fernandez · runs 10+ AI cron jobs daily on a personal Linux server

The gap between "I have an AI agent" and "I have an AI agent that actually does things" is a cron job.

You can have the best memory system in the world, the best prompts, the best tool access. If you still have to open a terminal and start the thing every time, it's not doing much for you. Scheduling is what makes it useful.

Cron jobs are how you fix that. They're how your morning briefing shows up on your phone before you're out of bed. How your JIRA queue gets reviewed without you touching it. How your Spanish practice lesson arrives whether you remembered to ask for it or not.

This post walks through the setup: job configurations, model choices, and what it actually costs to run.

What a Cron Job Actually Is

A cron job is a scheduled task. Your system runs it at the time you specify, whether you're there or not.

If you've set a recurring calendar reminder, you understand the mechanism. The difference is: instead of reminding you to do something, the agent goes and does it.

Syntax looks like this:

0 6 * * * /path/to/my-morning-briefing.sh

That runs at 6:00am every day. Five fields: minute, hour, day-of-month, month, day-of-week. Asterisk means "every." If you want to run something at 9am Monday through Friday only: 0 9 * * 1-5.

You set these up in a file called the crontab. Run crontab -e to edit it. That's it. Linux, macOS, WSL2, all support it.

What Makes a Good Cron Job Candidate

Not every task should be a cron job. The ones that work well share three characteristics:

A clear, bounded task. "Review JIRA tickets that haven't been touched in 48 hours and flag ones that need my attention" is good. "Do my work" is not. The job needs to know when it's done.

Output that goes somewhere useful. If the agent runs but you have no idea what it produced, you won't trust it. My cron jobs all deliver output to Telegram. I can see exactly what ran and what it decided.

No interactive clarification required. A cron job can't ask you a follow-up question. It needs enough context in its setup to handle the task without a back-and-forth. If a task normally takes five clarification questions before you can start, it's not a good cron job yet. Fix the prompts and context first.

My 10 Cron Jobs

Here they are, with schedule, model, approximate token cost per run, and what they actually do.

1. Morning Briefing

Schedule: 6:00am daily
Model: Sonnet
Cost per run: ~$0.08

Pulls news on topics I care about: AI, Tesla (I work there), Texas BBQ, and marathon running. Compiles a digest. Sends to Telegram.

By the time I'm making coffee, I already know if anything relevant happened overnight.

The model choice matters here. This job generates output, so it goes to Sonnet. I want something well-written that I'll actually read. Haiku is fine for reads and checks. For output I consume, Sonnet.

2. Memory Maintenance

Schedule: 6:15am daily
Model: Haiku
Cost per run: ~$0.02

Reads that day's conversation logs. Extracts anything worth keeping into my daily notes file. Compresses older notes that are no longer active. Updates MEMORY.md if anything has changed significantly.

Haiku here because this is a read/extract/compress task. No original generation needed. Haiku is four to five times cheaper than Sonnet for the same input size, and it's fast enough that this job finishes before the next one starts.

This job is the reason my memory system doesn't grow unbounded. Without it, MEMORY.md would eventually get large enough to eat a significant portion of my context window. This keeps it trim.

3. Spanish Practice - Morning

Schedule: 10:00am weekdays
Model: Haiku
Cost per run: ~$0.01

Sends me a Spanish vocabulary or grammar exercise. Specific to my current level and what I've been working on (tracked in memory files).

Three Spanish crons throughout the day, each one slightly different in format. The fact that they just show up is what makes the practice stick. When I had to opt-in, I'd skip. When it appears on my phone, I do it.

Haiku because this is a templated quiz task. The content is simple enough that I don't need a more capable model.

4. Spanish Practice - Midday

Schedule: 1:00pm weekdays
Model: Haiku
Cost per run: ~$0.01

Conversational exercise. I reply, the agent responds, I get feedback on my Spanish. Short loop, low overhead.

5. Spanish Practice - Evening

Schedule: 7:00pm daily
Model: Haiku + ElevenLabs
Cost per run: ~$0.03 (Haiku) + ~$0.05 (TTS)

Lesson with a voice component. Haiku generates the lesson content. ElevenLabs converts it to audio. I can hear the pronunciation, not just read it.

The TTS add-on is worth it for language learning specifically. Reading "buenas tardes" and hearing it are different experiences. I use a Spanish-trained ElevenLabs voice. English-trained voices sound wrong in Spanish (I made that mistake early on).

6. Health Check

Schedule: 9:00am daily (plus spot checks throughout the day)
Model: Haiku
Cost per run: ~$0.01

Reads the heartbeat timestamps that my other cron jobs write when they run. If any job has been silent longer than expected, sends a Telegram notification immediately.

This is the most important job I have. Everything else depends on it.

Before I built this, I found out about failures when I noticed the morning briefing hadn't shown up. That could be twelve hours after the failure. Now I know within an hour.

Haiku because this is pure data reading and conditional alerting. No generation. Fast and cheap.

7. JIRA Queue Review

Schedule: 9:30am weekdays
Model: Sonnet
Cost per run: ~$0.12

Polls my JIRA queue for tickets that haven't been reviewed in the last 24 hours. Runs each one through a structured review: is the information complete? What category does it fall into? Does it need my attention, or is it something the standard process handles?

Flags what needs me. Handles what doesn't.

Before agents, I reviewed every ticket manually. This is the job that most clearly shows the ROI. Not because it saves money. Because it removes the mental overhead of reviewing the full queue when most of it doesn't need my eyes on it.

Sonnet because the review requires actual judgment. I want the output to hold up. This isn't a template task.

8. Knowledge Consolidation

Schedule: 8:00pm weekly (Sunday)
Model: Sonnet
Cost per run: ~$0.15

Reads the week's daily notes. Pulls out decisions made, things learned, projects that moved. Writes a weekly summary to a dated file.

The weekly summary is what I actually review on Monday mornings before the work week starts. The daily notes are too granular for that. The weekly roll-up is the right level of detail.

9. Daily Summary and Memory Write

Schedule: 11:00pm daily
Model: Sonnet
Cost per run: ~$0.10

End-of-day review. What happened today, what to carry into tomorrow, any updates to MEMORY.md or the project files.

This and the morning briefing are bookends. The morning briefing gets me oriented. The daily summary closes the loop.

10. Token Report

Schedule: 11:30pm daily
Model: Haiku
Cost per run: ~$0.01

Reads the day's API usage logs. Generates a brief token spend report: total tokens, cost by model, cost by job type. Appends to the weekly usage file that my token dashboard reads.

Without this, you're flying blind on costs. With it, you see which jobs are expensive, which models are getting overused, and where to optimize.

The Model Routing Math

Let me make the cost difference concrete.

Before I set up model routing, I was running most jobs on Sonnet. Here's what the same 10-job schedule costs at different model assignments:

Scenario Daily Cost
Everything on Sonnet ~$0.90/day
Everything on Opus ~$4.50/day
Routed (Haiku for maintenance/checks, Sonnet for generation) ~$0.35–0.50/day

That's the pure cron job cost. Add interactive sessions and you get to my ~$3/day total.

The routing rule I use: if the job generates output I'll consume or that requires quality, use Sonnet. If it reads, checks, compresses, or sends a simple formatted notification, use Haiku. Opus when complexity of reasoning actually justifies it (which is rarely for scheduled tasks).

Common Failure Modes

The job runs but you can't tell. Add output delivery to every job. Telegram notification, file write, something you can inspect. If the only indication that a job ran is a log file you never check, you won't know when it's broken.

Timezone mismatch. This one got me. Some of my jobs were running in local time, some in UTC. That caused downstream failures because the expected input wasn't ready at the expected time. Be explicit about timezone in every cron config. TZ=America/Chicago at the top of your crontab.

Context going stale. The job still runs but produces worse output because MEMORY.md hasn't been updated in a week or the prompt references something that changed. The health-check cron catches outright failures. For quality drift, do a manual spot check periodically.

Rate limiting. If you have several cron jobs firing in close succession, you might hit API rate limits. I stagger my jobs by at least 15 minutes. The 6:00am briefing and 6:15am memory maintenance don't conflict because there's buffer.
