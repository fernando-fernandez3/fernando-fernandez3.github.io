How to Monitor Your AI Agents So You Know When They Break

Secondary keywords: AI agent reliability, cron job monitoring, Claude agent health check
Search intent: Developer whose agent broke silently and they didn't find out until days later.
Target word count: 1,800-2,200 words
SEO outline note: This is Chapter 7 of the free guide, expanded as a standalone post.

The first time my agent broke silently, it was running for two days before I noticed. A cron job stopped producing output. No error message, no notification, nothing. I checked my Telegram feed and realized: it had been quiet. Too quiet.

The problem with autonomous agents is that "it worked yesterday" gives you zero information about today. An agent that ran fine for three weeks can silently stop at any point: API key expired, context window exceeded, a cron typo, a dependency that got updated. You won't know unless you build monitoring in from the start.

Here's what I run. It's not complicated, but it works.

The two layers of monitoring

There's a difference between knowing that your agent ran and knowing that it ran correctly.

Layer 1: Heartbeat monitoring. Did the job fire? Did it finish? A heartbeat is a timestamp written at the end of a successful run. If the expected heartbeat is missing, something went wrong.

Layer 2: Output validation. Did the job produce something useful? A job can complete successfully and still output garbage. This layer checks quality, not just completion.

Most setups only need Layer 1. If your cron produces a file or sends a message, you can infer success from the presence of output. Layer 2 is for more critical tasks where "ran" and "ran correctly" are genuinely different.

Start with Layer 1. Add Layer 2 if you have a specific job where it matters.

Layer 1: Heartbeat timestamps

The pattern is simple. At the end of every cron job, write a timestamp to a file:

echo "$(date -u +%Y-%m-%dT%H:%M:%SZ)" > /tmp/agent-heartbeats/morning-briefing.txt

A separate monitoring job (I run mine every hour) reads those files and checks whether they're fresh:

import os
from datetime import datetime, timezone, timedelta
from pathlib import Path

HEARTBEAT_DIR = Path("/tmp/agent-heartbeats")
MAX_AGE_HOURS = {
    "morning-briefing": 26,   # fires at 6am, alert if missing by 8am next day
    "memory-maintenance": 26,
    "daily-summary": 26,
    "health-check": 2,        # fires every hour, alert if missing for 2 hours
}


def check_heartbeats() -> list[str]:
    """Returns a list of stale or missing heartbeats."""
    alerts = []
    now = datetime.now(timezone.utc)

    for job_name, max_hours in MAX_AGE_HOURS.items():
        heartbeat_file = HEARTBEAT_DIR / f"{job_name}.txt"

        if not heartbeat_file.exists():
            alerts.append(f"{job_name}: no heartbeat file found (never ran?)")
            continue

        last_beat_str = heartbeat_file.read_text().strip()
        try:
            last_beat = datetime.fromisoformat(last_beat_str)
        except ValueError:
            alerts.append(f"{job_name}: invalid heartbeat timestamp: {last_beat_str!r}")
            continue

        age = now - last_beat
        if age > timedelta(hours=max_hours):
            hours_ago = round(age.total_seconds() / 3600, 1)
            alerts.append(f"{job_name}: last ran {hours_ago}h ago (max allowed: {max_hours}h)")

    return alerts

If there are alerts, send them to Telegram. If there are none, stay quiet:

import httpx
import os

def notify(message: str) -> None:
    token = os.environ["TELEGRAM_BOT_TOKEN"]
    chat_id = os.environ["TELEGRAM_CHAT_ID"]
    httpx.post(
        f"https://api.telegram.org/bot{token}/sendMessage",
        json={"chat_id": chat_id, "text": message},
    )


alerts = check_heartbeats()
if alerts:
    notify("⚠️ Agent health check:\n" + "\n".join(f"• {a}" for a in alerts))

The health check job itself writes a heartbeat. Which brings up the obvious problem.

The watcher's watcher problem

If your health check job fails, who tells you?

This is a real issue. The health checker is also a cron job. It can also break silently.

Two approaches:

Option 1: Two-tier monitoring. Your health checker monitors the other jobs. A separate, lightweight script (maybe a system cron, not an agent cron) monitors the health checker. The system cron can send you a simple ping via curl to a free uptime monitoring service like UptimeRobot or Betterstack. Those services alert you if the ping stops arriving.

Option 2: Dead man's switch. Set up a free service like healthchecks.io. It expects a ping on a schedule. If the ping doesn't arrive, it notifies you. Your health check job pings healthchecks.io at the end of every successful run. If the job breaks, the ping stops, and you get an email or text.

For a personal setup, Option 2 is simpler. Healthchecks.io has a generous free tier. One ping URL per job is all you need.

# At the end of your health check cron:
curl -fsS --retry 3 https://hc-ping.com/your-unique-uuid-here

What to put in your heartbeat files

A timestamp is the minimum. You can add more if it's useful:

2026-03-09T14:00:05Z | duration_s=3.2 | output_chars=847 | model=claude-haiku-4-5

Then your monitoring job can check output size (zero characters = something went wrong) or duration (way longer than usual = probably stuck or looping).

Don't over-engineer this. A bare timestamp is usually enough. Add fields when you have a specific failure mode you're trying to detect.

Alerting: what to tell you vs. what to log quietly

Not every issue should wake you up.

My rule: if a job missed one run, log it. If it's missed two consecutive runs, alert. One miss could be a transient network issue or a brief API outage. Two misses in a row is a real problem.

CONSECUTIVE_MISS_THRESHOLD = 2

def should_alert(job_name: str, consecutive_misses: int) -> bool:
    return consecutive_misses >= CONSECUTIVE_MISS_THRESHOLD

For jobs that run every hour, one miss isn't worth a notification. For jobs that run once a day, any miss is worth knowing about.

Adjust thresholds per job. A morning briefing missing two days in a row is a problem. A health check missing two consecutive hourly runs is a problem. A weekly review missing one week is annoying but not urgent.

Token budget monitoring

An agent that's running correctly but burning through your API budget at 10x the expected rate is still a problem. Token monitoring is separate from job monitoring, but it belongs in the same health check.

The Anthropic API usage endpoint gives you cost data by model and time period. Poll it daily and compare against your expected budget:

import httpx
import os

def get_daily_token_cost() -> float:
    """Fetches today's API cost from Anthropic usage endpoint."""
    # Anthropic's usage API - check docs.anthropic.com for current endpoint
    headers = {"x-api-key": os.environ["ANTHROPIC_API_KEY"]}
    response = httpx.get(
        "https://api.anthropic.com/v1/usage",
        headers=headers,
        params={"granularity": "day"},
    )
    response.raise_for_status()
    data = response.json()
    # Extract today's total - structure varies, check the current API response format
    return data.get("total_cost_usd", 0.0)


DAILY_BUDGET_USD = 5.00
cost = get_daily_token_cost()
if cost > DAILY_BUDGET_USD:
    notify(f"⚠️ API cost today: ${cost:.2f} (budget: ${DAILY_BUDGET_USD:.2f})")

Running this check daily surfaces runaway jobs before they become expensive. I've caught a cron that had an infinite retry loop (it was hammering the API trying to recover from a bad response) this way.

The token dashboard

For visibility without constant alerts, I have a simple HTML dashboard that pulls data from log files and shows token spend by job and by model. It lives on my local network and I check it once a week.

The token-dashboard.html in the Product #1 bundle does this with no backend: it reads from a log file you write during each cron run. It's ugly but useful.

The key fields to log per cron run:
- Timestamp
- Job name
- Model used
- Input tokens
- Output tokens
- Estimated cost

If your agent framework doesn't expose token counts, some providers log them in response headers or response metadata. Claude's API includes token usage in every response object.

Setup checklist

Here's the full monitoring setup in order:

  1. Heartbeat files. Add one line to the end of each cron job that writes a timestamp.
  2. Health check cron. Runs every hour. Reads heartbeat files, sends Telegram alerts on stale/missing heartbeats.
  3. Dead man's switch. Set up healthchecks.io (free tier). Health check cron pings it on success.
  4. Token budget check. Runs daily. Pulls API usage and alerts if over budget.
  5. Dashboard. Optional. Gives you a weekly review view. Bundle has a starter version.

That's it. None of this is complicated. The hardest part is remembering to add the heartbeat write to each new cron job you create. Make it a habit: new cron gets a heartbeat line before you ship it.

The logs

Keep logs. Every cron job should write to a log file: job start, job end, any warnings or errors. Standard Python logging is fine:

import logging

logging.basicConfig(
    filename=f"/var/log/agents/{job_name}.log",
    level=logging.INFO,
    format="%(asctime)s %(levelname)s %(message)s",
)

Rotate them weekly. logrotate handles this on Linux. Unrotated logs will eventually fill your disk. I found this out the hard way on a setup I left running for four months without checking.

What you actually need vs. what sounds good

The monitoring setups I've seen people build are often way too complex. Grafana dashboards, Prometheus metrics, custom alerting pipelines. For a personal agent setup with 5-15 cron jobs, that's overkill.

What you actually need:
- Heartbeat timestamps written by each job
- A health check cron that reads them
- One external ping service to catch health check failures
- Telegram for alerts

What you don't need:
- Vector databases for log storage
- ML-based anomaly detection
- A dedicated monitoring server

Keep it proportional to the scale. A personal agent system that saves you time and costs $3/day doesn't need $50/month in monitoring infrastructure.
