How to Build a Production AI Agent System (Real Architecture, Real Costs)

Fernando Fernandez 11 min read 2,064 words
By Fernando Fernandez · Staff Support Engineer at Tesla, self-taught Python dev, runs 10+ AI cron jobs daily

Most AI agent tutorials end the moment the demo works.

You paste in an API key, the agent responds in the terminal, and the tutorial says "great, you've built an agent." But what they showed you is a chatbot with extra steps. You still have to start it. You still have to remember to check it. The moment you close the terminal, it's gone.

A production agent is different. It runs at 6am while you're asleep. It remembers what you told it three weeks ago. It tells you when it breaks. It costs about the same as a lunch every two days.

This post is about building that. Architecture, costs, and failure modes from my experience.

The Four Problems That Break AI Agents in Production

Before I built something that worked, I built several things that didn't. The problems kept showing up in the same four flavors.

1. Memory Loss Between Sessions

The first week I ran my agent, I'd have a great conversation. We'd work through something, land on a conclusion, make a plan. Then the next morning I'd pick up where I left off and my agent would say something like, "I'm not sure what you're referring to, could you explain?"

Every session starts blank. The API is stateless by design. You're not forgetting to save something. The architecture just doesn't persist anything unless you build that layer yourself.

People try a few workarounds. Some paste previous chat history back into context. That works for a few sessions before the context fills up and gets expensive. Some try plugins or vector databases. Those work, but they add setup and maintenance overhead that's hard to justify for a personal system.

There's a simpler approach. I'll get to it.

2. No Visibility Into What the Agent Did

An agent ran for three hours. You have no idea what it actually did. Was it making progress? Did it hit an error and silently loop? Did it complete the task ten minutes in and spend the rest of the time doing nothing?

This is the "quis custodiet" problem, who watches the watcher. Without logging and monitoring, you're flying blind. You find out something broke when you notice the output stopped showing up, not when it actually happened.

3. Unpredictable Token Costs

Tokens are the unit that AI models bill you on. Roughly three-quarters of a word. When your agent reads a file, that's tokens. When it writes a response, that's tokens. Long context histories compound fast.

The naive implementation routes everything to your most capable (most expensive) model. A morning news digest doesn't need the same model you'd use for complex analysis. Running everything on your heavy model is like hiring a principal engineer to write your meeting notes. Wasteful.

I've seen my setup cost $30/day when I was doing it wrong. After fixing model routing, same workload, same number of jobs: $3/day.

4. Constant Babysitting

"Autonomous" agents that still need you to initiate them aren't autonomous. They're just a fancier way to paste a prompt.

The goal is an agent that runs its morning briefing at 6am without you touching anything. That emails you a summary while you're at the gym. That alerts you when something breaks without you having to notice it's broken first.

That requires scheduling. And scheduling requires a slightly different setup than most tutorials show.

The Architecture That Solves All Four

Four components. Each one targets a specific problem. In order of how much pain you'll feel if you skip one:

  1. Memory layer - solves context loss
  2. Cron jobs - solves the "requires babysitting" problem
  3. Monitoring - solves visibility
  4. Token optimization - solves unpredictable costs

Component 1: Memory Architecture

The three-file system is the core of mine. No database. Just markdown files on disk.

MEMORY.md is long-term context. Name, current projects, ongoing decisions, preferences, things that shouldn't require re-explaining every session. My agent reads this at the start of every session. If I've been troubleshooting my pool pump for two weeks, that's in there. It's the context layer that makes conversations feel continuous instead of starting from zero.

Daily notes are episodic memory. Each night, a cron job goes through that day's conversations and extracts what matters. Not transcripts. More like a debrief. "Fernando mentioned the pool pH is running high. He wants to look at it this weekend. Marathon training is going well, he hit his pace target." Gets written to a dated file. If I need to know what I decided about something three weeks ago, it's there.

AGENTS.md is the per-agent instruction file. Every agent reads it at session start. It describes what the agent does, what tools it has access to, and how it should behave. This is how you keep multiple agents from conflicting with each other.

Three files, no external dependencies. If something breaks, I open the files in a text editor and see exactly what's going on.

The maintenance piece matters: as daily notes accumulate, the memory grows unbounded. I have a nightly cron job that runs a compression pass using Haiku (cheap and fast for this kind of read/summarize task). It pulls what's still relevant into MEMORY.md and archives the rest. Context stays manageable.

What breaks without this: everything gets worse. Your agent can't reference last week's decision. You repeat yourself constantly. The more agents you add, the worse it gets because they're all missing the same context. Build memory first.

Component 2: Cron Job Architecture

A cron job is a scheduled task. If you've ever set a recurring calendar reminder, you understand the concept. Same idea, but instead of reminding you to do something, your agent goes and does it.

This is what separates an agent from a chatbot. A chatbot waits for you. An agent runs whether you show up or not.

My 10 cron jobs, organized by type:

Briefings and digests
- 6:00am: Morning news digest on AI, Tesla, running, and Texas BBQ. Delivered to my phone via Telegram before I'm out of bed.
- 11:00pm: Daily summary. What happened, what to carry into tomorrow, memory write.

Recurring tasks
- Three times a day: Spanish practice lessons. Because when it just shows up automatically, I do it. When I had to opt-in, I'd skip it.

Maintenance jobs
- 6:15am: Memory compression and maintenance. Haiku reads that day's notes, updates MEMORY.md, archives what's stale.
- 9:00am: Health check. Verifies the other jobs ran. Reports failures immediately.

Work-specific jobs
- Periodic JIRA queue review. Checks for unreviewed tickets, flags what needs my attention, handles what doesn't. Before agents, I reviewed every ticket manually.

The model routing piece is where most people leave money on the table. My config specifies a model per job:

  • Haiku: health checks, file reads, memory compression, Spanish quizzes. Cheap, fast, good enough.
  • Sonnet: morning briefings, daily summaries, anything with output quality requirements.
  • Opus: complex reasoning, multi-step analysis (roughly 5% of my total calls).

Before I set this up, I was running everything on Sonnet. That alone dropped my costs by around 60% when I switched.

The analogy I use: if you're a company and you send your principal engineer to grab coffee, you're misallocating. That's what a junior hire is for. Same logic applies here. Match the model to the task, not the other way around.

Component 3: Monitoring and Reliability

Here's what happens without monitoring: you set up a cron job, it runs fine for two weeks, then quietly starts failing. You don't notice. Your morning briefing stops showing up and you think you've just been busy. Three days later you go digging.

That happened to me. More than once.

The health-check cron

Each of my cron jobs writes a heartbeat timestamp when it runs. A separate health-check cron runs every hour. It reads those timestamps. If any job has been silent for longer than expected, it sends a Telegram notification immediately. Not the next morning when I notice the briefing didn't show up. Immediately.

This is two layers:

  1. Each job writes "I ran at 6:02am"
  2. The health-check reads those entries and says "job X should have run by 6:30am and hasn't"

If the health-check itself fails, that's a different alert path. The point is: you need to watch the watcher.

Failure modes I've actually hit

Timezone mismatch. Parts of my setup were running in local time, parts in UTC. That caused a cron to trigger at the wrong time, which caused downstream jobs to fail because the expected input wasn't ready. Not obvious. Took a while to find. Now I'm explicit about timezone in every job config.

Silent failures. Before I built the health check, jobs would fail with no notification, no error visible anywhere. The job just didn't run. Build monitoring before you need it. It takes less time to set up than to debug a week of silent failures.

Quality drift

The health check catches outright failures. Quality drift is harder. A cron might still be running but producing worse output because the context has gotten stale, a prompt has drifted, or an upstream data source changed. For this, I do a periodic manual review. Not daily. But I do check. Automation reduces oversight, it doesn't replace it.

Component 4: Token Cost Optimization

Real numbers from my setup:

  • Daily cost: ~$3 for 10+ cron jobs plus interactive sessions
  • Model split: ~60% Haiku, ~35% Sonnet, ~5% Opus
  • Monthly plan: Anthropic Max at $100/month. The only limit I've hit is rate limiting on requests, not token budget.

The token dashboard is an HTML file that visualizes usage by model and job type. When I saw which jobs were burning the most, I knew where to optimize. You can't manage what you can't see.

Context trimming matters too. My agents don't load the full memory archive every session. They load MEMORY.md plus the relevant daily note. If they need more context, they know where to look. Loading everything by default is the same mistake as defaulting to Opus for everything: expensive and unnecessary.

The decision tree I use for model selection:

  • Does the task require multi-step reasoning or generate something that has to be high quality? Sonnet minimum.
  • Is it a read/check/summarize/compress task? Haiku.
  • Is it something genuinely complex where quality of reasoning matters? Opus, but rarely.

Real Numbers

Setup as of early 2026:

  • 10 cron jobs, various schedules throughout the day
  • Telegram for all notifications (personal and family group)
  • ElevenLabs for Spanish TTS, Kokoro for English TTS (local, zero cost)
  • Memory layer: MEMORY.md + dated daily notes + topic files by conversation type
  • $100/month on Anthropic Max

Time investment: maybe 30 minutes a day on average. Some days zero. Some days two hours if I'm adding something or debugging something weird. The maintenance overhead is low once the system is stable.

What surprised me most: the family coordination piece. I set it up expecting it to be a nice-to-have. My wife is in a group chat with me and my agent. Three days before a family trip, the night before, morning of, my agent sends reminders. Neither of us forgets. That turned out to be the most consistently valuable thing the system does.

What I'd do differently: build the memory system first, before anything else. I lost two weeks of useful context because I didn't. Build the health-check cron before anything else, for the same reason.
