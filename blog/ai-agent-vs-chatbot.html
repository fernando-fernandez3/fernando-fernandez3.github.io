What's the Difference Between an AI Chatbot and an AI Agent?

Fernando Fernandez 8 min read 1,513 words
By Fernando Fernandez · Staff Support Engineer at Tesla, runs a production AI agent system daily

The short answer: a chatbot waits. An agent acts.

That's not a marketing distinction. There's a real architectural difference between "Claude in a browser tab" and "Claude running on a schedule with access to your files, your email, and your task manager." Same model, completely different capability.

I've been running AI agents in production for a while now: daily briefings, JIRA queue review, Spanish practice, family coordination. None of what I use them for is possible in a chatbot. This post is about why, and what it actually takes to cross that line.

What a Chatbot Does

A chatbot responds to you.

You open it, type something, it replies. The conversation ends or continues. When you close the tab, nothing happens. When you open it again tomorrow, it doesn't remember yesterday unless the platform stored the history and included it in context automatically.

ChatGPT in a browser is a chatbot. Claude.ai in a browser is a chatbot. Extremely capable ones. But they're reactive by design (they respond when prompted, not before).

The limitation isn't the model's intelligence. The same model that powers those browser interfaces can power an agent. The limitation is everything around the model: no memory between sessions, no way to schedule it, no access to your actual environment.

What an Agent Does

An agent completes tasks. Usually without you initiating them.

Here's a concrete example. I work at Tesla as a Staff Support Engineer. My team gets JIRA tickets from stakeholders: feature requests, issue reports, questions. Some tickets have everything I need. Some are incomplete or low priority. All of them need to be looked at.

Before agents: I reviewed every ticket manually. It took time and it was mostly routine triage work.

Now: I have an agent that polls the JIRA queue on a schedule. It looks at each unreviewed ticket, checks for completeness, assesses priority based on criteria I've defined, and flags the ones that actually need my attention. The rest get an initial response or a note for when I'm ready.

I didn't start that session. I didn't prompt anything. I woke up, the work was done.

That's the thing a chatbot can't do. Not because it's less capable at the actual reasoning (it might be identical). Because it has no way to run without me showing up and asking it to.

The Three Things That Make an Agent

Three things make something an agent.

Memory

A chatbot's memory resets every session. Whatever happened yesterday, whatever you decided two weeks ago, whatever context matters for understanding your situation: it's gone when you close the tab.

An agent needs persistent memory. Not in the "the platform saved your chat history" sense. In the "the agent has a structured record of what matters, can search it, and actively uses it to do better work" sense.

My setup uses flat files. Three of them:

MEMORY.md is the working memory. It has my name, my preferences, my current projects, the things I'd have to re-explain every session if it didn't exist. The agent reads this at the start of most sessions.

Daily notes are episodic. Every night, a cron job reviews all my conversations from that day and writes a summary: what I was working on, what decisions I made, what's coming up. Dated files. The agent can read recent ones for context without loading everything.

Topic files are focused. I use Telegram topics to organize conversations by subject (work, family, fitness, etc.). Notes from those conversations get organized by topic so I can find past decisions without digging through a general log.

The flat file approach is deliberate. No database, no vector store, no external service required. Everything is readable, versionable, and survives a system restart. For personal-scale memory, it's all I need.

Scheduling

A chatbot runs when you run it. An agent runs on a schedule.

This sounds simple. The implications are not.

If your agent needs to run at 6am every day, you either need to be awake at 6am to start it, or you need a cron job. Cron jobs are how Unix-based systems schedule recurring tasks. You define a schedule, you define a command, and the system runs it whether you're there or not.

My 10 cron jobs cover: morning briefing, three Spanish lessons throughout the day, memory maintenance each night, health checks, and a few work-specific things. I don't think about any of them. They run on their schedules. The output shows up on my phone.

Without scheduling, you're still the one kicking off every interaction. The whole value of an agent is that it runs on its own.

Tool Access

A chatbot can talk about your email. An agent can read it.

Tool access is what gives an agent the ability to reach into your actual environment: your files, your calendar, your APIs, your task manager. The agent doesn't just discuss what you should do. It does it.

My agents can read files on my machine, write to my memory files, call external APIs, send Telegram messages, make HTTP requests. When I'm not around, they're working with real data and producing real output. Not describing how they would do something if they had access. Actually doing it.

Most agent frameworks give you a way to define tools and expose them to the model. Claude Code has direct file system access out of the box. Other setups use function calling: you define what functions the agent can call, the model decides when to call them, and your code handles the execution.

Why All Three Matter Together

The reason "all three" is the answer and not just one of them: they compound.

Memory without scheduling means you have context but you still have to show up every time. Scheduling without memory means your agent runs but starts from scratch each session. Tool access without scheduling means you have a capable chatbot that still requires manual initiation. Tool access without memory means the agent can act but can't improve over time because it doesn't know what worked before.

All three together is what produces the "it just handled that" feeling. The agent ran, had the right context, had access to what it needed, and delivered output (without you starting it).

The Realistic Starting Point

If you've never built an agent before, don't try to build all three at once. Here's what I'd do differently if I were starting over:

Week 1: Memory system only. Before you try to schedule anything or give the agent tool access, set up the three-file memory system. Run sessions manually. Make sure the context is working. Make sure yesterday's decisions are informing today's work. This is the foundation.

Week 2: One cron job. Pick the most obvious candidate: a daily briefing, a weekly review, something you do manually and routinely. Configure one cron job. Make sure it produces output you actually want to read. Don't build ten at once.

Week 3: Expand. Once you have memory working and one job running reliably, the pattern is established. Add more crons. Add tools. Wire it up to Telegram so the output reaches your phone.

I skipped building memory first and spent two weeks losing context that should have been captured. I didn't build health monitoring early enough and spent time debugging things that should have been immediately visible. Starting with memory and one reliable cron would have saved me both.

What You Can Build Once You Have This

The architecture is general. Once you have memory, scheduling, and tool access working together, the question is just: what manual work am I doing on a regular basis that fits the pattern?

My JIRA triage. My morning briefing. My Spanish practice. Family coordination reminders. I've got a call screener in progress that takes incoming calls, handles routine inquiries, and only escalates to me when something actually needs my attention.

All of it is the same pattern, just applied to different problems.

The chatbot version of AI is genuinely useful. The agent version is different in kind, not degree. You go from "I can use this as a tool" to "this runs part of my life and I barely think about it."
